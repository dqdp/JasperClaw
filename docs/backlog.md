# Backlog

## Purpose

This backlog translates the roadmap into epics and tasks that can be tracked in a project tool such as Linear.

This version of the backlog reflects the architecture decisions accepted in ADR 0004 through ADR 0013.

## EPIC 1: Control Plane MVP

### Outcome

`agent-api` replaces the current stubbed text path and becomes a real OpenAI-compatible control plane backed by `Ollama` and `Postgres`.

This is the first real vertical slice for v1.

### Tasks

- CP-1: Split `agent-api` into `routers`, `schemas`, `services`, `clients`, and `config`
- CP-2: Define Pydantic schemas for `models`, `chat/completions`, `audio/transcriptions`, and `audio/speech`
- CP-3: Implement `Ollama` client with timeout, retry policy, and explicit error mapping
- CP-4: Replace stubbed `POST /v1/chat/completions` with real non-streaming model execution
- CP-5: Add SSE streaming support for `POST /v1/chat/completions`
- CP-6: Implement logical profile routing for `assistant-v1` and `assistant-fast`
- CP-7: Add `Postgres` integration and persist conversations, messages, requests, and model runs
- CP-8: Make `GET /readyz` depend on live downstream checks
- CP-9: Add request IDs and structured logging across the request path
- CP-10: Add integration tests for `agent-api -> Ollama -> Postgres`

### Exit criteria

- Open WebUI can complete a text chat through `agent-api`
- responses are generated by the runtime model
- persistence works for at least one full conversation
- the text path is observable and readiness-aware
- failures return explicit machine-readable errors
- the text path is real, persisted, observable, and failure-stable

## EPIC 2: Memory Foundation

### Outcome

The assistant can retain useful state across sessions and retrieve relevant memory during orchestration.

### Tasks

- MEM-1: Design canonical `Postgres + pgvector` schema for memories and embeddings
- MEM-2: Add embedding generation path for user and assistant messages
- MEM-3: Store memory candidates with source metadata and timestamps
- MEM-4: Implement retrieval query path in `agent-api`
- MEM-5: Inject retrieved context into prompt assembly
- MEM-6: Add memory retention, invalidation, and deletion rules
- MEM-7: Add evaluation fixtures to verify retrieval relevance
- MEM-8: Add integration tests for persistence and retrieval behavior

### Exit criteria

- relevant memory is retrieved during later conversations
- retrieval logic is observable and auditable
- deletion and retention rules are explicit

## EPIC 3: Tools Integration Layer MVP

### Outcome

All tool usage runs through a typed internal integration boundary inside `agent-api`, with consistent logging and policy enforcement.

### Tasks

- TG-1: Define shared request and response envelope for internal tool execution
- TG-2: Implement `POST /tools/web-search`
- TG-3: Add placeholder but stable contracts for Spotify endpoints
- TG-4: Add timeout, retry, and dependency error mapping rules
- TG-5: Implement tool audit logging in `agent-api`
- TG-6: Add tool policy layer for allow and deny decisions
- TG-7: Add integration tests covering success, timeout, and denial paths

### Exit criteria

- tool calls happen only through the typed tools integration boundary owned by `agent-api`
- tool executions are logged with request context
- failures are explicit and bounded

## EPIC 4: Voice Path

### Outcome

Voice becomes a second-order v1 feature through the same orchestration core as text, but only after the text path is stable.

### Tasks

- VP-1: Implement `POST /transcribe` in `stt-service`
- VP-2: Implement `POST /speak` in `tts-service`
- VP-3: Normalize transcription and speech requests in `agent-api`
- VP-4: Persist voice interactions in the same canonical store as chat
- VP-5: Add tests for transcription and synthesis proxy flows
- VP-6: Add end-to-end smoke coverage for voice endpoints

### Exit criteria

- voice flows through `agent-api`
- voice requests share memory and policy behavior with text
- STT and TTS failures are observable and recoverable

## EPIC 5: Operations and Hardening

### Outcome

The platform becomes repeatable to deploy, observe, and roll back.

### Tasks

- OPS-1: Define all application env vars and defaults in one configuration document
- OPS-2: Add structured logs for every service
- OPS-3: Add service-level metrics and dependency timing
- OPS-4: Expand smoke tests to cover readiness, chat, and key internal contracts
- OPS-5: Add database migration workflow
- OPS-6: Define backup and restore procedure for `Postgres`
- OPS-7: Validate rollback procedure against deployed image versions
- OPS-8: Pin production runtime versions and document upgrade policy
- OPS-9: Enforce layered lint, test, and smoke gates across local, CI, and deployment flow

### Exit criteria

- deployment is reproducible
- rollback is documented and tested
- key operational signals are visible without ad hoc debugging

## Recommended execution order

1. EPIC 1: Control Plane MVP
2. EPIC 2: Memory Foundation
3. EPIC 3: Tools Integration Layer MVP
4. EPIC 4: Voice Path
5. EPIC 5: Operations and Hardening

## Execution rules

- EPIC 1 is the first real delivery slice and should be complete before EPIC 4 begins
- no task may bypass the canonical request path through `agent-api`

## Notes for Linear import

- each epic can map to a project milestone or parent issue
- each task ID is stable enough to reuse in branch names and commit messages
- if needed, EPIC 2 and EPIC 3 can run partially in parallel after EPIC 1 is stable
